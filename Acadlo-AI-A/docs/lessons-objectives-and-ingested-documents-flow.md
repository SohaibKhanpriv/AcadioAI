# Lessons, Objectives, and Ingested Documents — Flow and Gaps

This document explains how lessons and objectives are created, how they relate (or don’t yet) to ingested documents (e.g. a book), and how the tutor is supposed to know which knowledge base to use. It ties the implementation to what the client/specs say.

---

## 1. How Lessons and Objectives Are Created Today

### 1.1 No Creation or Extraction Inside AI Core

In the current design, **AI Core does not create or extract lessons or objectives** from ingested content.

- **Doc 5 (M4 clarifications):**  
  - *"The caller (Acadlo backend) is responsible for supplying objective_ids in the intended order and supplying context_scopes."*  
  - *"AI Core does not call ABP/BE to discover objectives or scopes in v1."*

So in v1:

- **lesson_id** and **objective_ids** (or plain-English **objectives**) are **provided by the client** on `POST /v1/tutor/start`.
- The backend/Nuxt is responsible for:
  - Deciding which lesson the student is in.
  - Resolving which objectives belong to that lesson (e.g. from curriculum/ABP).
  - Sending them in the desired order.

### 1.2 Where They Enter the System

| Step | What happens |
|------|-------------------------------|
| Client | Calls `POST /v1/tutor/start` with `lesson_id`, `objective_ids` (or `objectives`), and optionally `context_scopes`, `ou_id`, `program_id`. |
| API | `StartTutorSessionRequest` validates that either `objective_ids` or `objectives` is present. If `objectives` is a list of plain-English strings, `_resolve_objectives()` turns them into stable IDs (e.g. `obj_adding_fractions`, `obj_equivalent_fractions`). |
| Runner | Builds `TutorStartParams` with `lesson_id`, `objective_ids`, `objective_labels`, `context_scopes`, etc. |
| Graph | `load_session_and_profile` **creates** a new `TutorSession` with these values and **creates** one `ObjectiveState` per `objective_id` (state = NOT_STARTED). No lookup from any “lesson registry” or ingested documents. |

So **lessons and objectives are created only in the sense of “session + objective states”**: the tutor creates a session and one objective state per ID the client sent. The **semantic meaning** of those IDs (e.g. “Chapter 3: Fractions”) lives in the client; AI Core just stores and uses the IDs and optional labels.

### 1.3 Optional: LLM-Generated Objectives (Progress Report)

The **adaptive-tutor-progress-report** describes an extension:

- When the client **does not** send a real lesson (e.g. `lesson_id = "pending"`), the tutor can:
  - Run an onboarding flow (topic, grade, level, language).
  - Use the collected **topic** (and grade/level) to **generate learning objectives and a lesson plan via an LLM**, then continue with the normal teaching flow.

So in that scenario, objectives are **generated by the tutor’s LLM**, not extracted from an ingested book. There is still no link to documents or chunks.

## 2. How Ingested Documents (e.g. a Book) Are Stored

The ingestion pipeline (Doc 1/2) is **separate** from the tutor’s lesson/objective model:

| Concept | Where it lives | Shape |
|--------|----------------|-------|
| **Document** | `documents` table | tenant_id, title, source_type, visibility_roles/scopes, tags, content_location_type/value, etc. |
| **Chunks** | `chunks` table | document_id, tenant_id, text, embedding, visibility_roles/scopes, tags, start_offset/end_offset |
| **Lesson** | Not in ingestion | No “lesson” or “curriculum” entity in the document/chunk schema. |
| **Objective** | Not in ingestion | No “objective” or “topic” entity in the document/chunk schema. |

So when you ingest “a book with 5 topics”:

- You get **one (or more) documents** and many **chunks** with embeddings.
- You can use **tags** (e.g. `{"topic": "fractions", "chapter": "3"}`) or **source_type** to distinguish content.
- There is **no** automatic extraction of “lessons” or “objectives” from the book; no table that says “this document is lesson X” or “these chunks are objective Y.” Any such mapping would have to be done by the caller (e.g. when sending `lesson_id` / `objective_ids`) or by a future feature that parses or tags content.

## 3. How the Tutor Uses (or Doesn’t Use) the Knowledge Base

### 3.1 context_scopes Are Stored but Not Used for RAG

- **TutorSession** and **TutorGraphContext** both have **context_scopes** (e.g. `["tenant", "ou:123", "class:7A"]`).
- Doc 4 (M4-A) and the code describe them as **“for RAG visibility filtering”** / “resolved visibility scopes for RAG/search calls.”
- In the **current implementation**, the tutor **never** calls the search or chat service. So **context_scopes are not used** to decide which knowledge base (which book/chunks) to query.

So today the tutor **does not** “know which knowledge base book to reference” in the sense of querying ingested documents. It only has:

- **lesson_id**, **objective_ids**, **objective_labels** (text), and **context_scopes** in context.
- Response generation builds the LLM prompt from **action_plan**, **lesson_context** (lesson_id, objective labels, states), and **objective_context** (current objective title, teaching state, etc.). No retrieved chunks are passed.

### 3.2 Where RAG Actually Exists

- **RAG** is implemented in **`/v1/chat`** (ChatService): it runs search (vector + tenant/roles/filters) and then LLM with retrieved chunks.
- **Search** accepts `tenantId`, `roles`, `query`, and optional **filters** (e.g. `sourceType`, `subject`, `grade`, `tags`). It does **not** accept `lesson_id` or `objective_id`; so even the generic chat API doesn’t filter by lesson/objective.

So:

- **Ingested documents** → used by **/v1/search** and **/v1/chat** (tenant + roles + optional filters).
- **Tutor** → uses **only** LLM + action plan + lesson/objective **labels**; no call to search or chat, so no link to ingested books.

## 4. What the Client / Spec Says About How It Should Be Done

### 4.1 Responsibilities Split (Doc 5)

- **Caller (Acadlo backend)** must:
  - Supply **objective_ids** (or **objectives**) in the intended order.
  - Supply **context_scopes** (e.g. `["tenant", "ou:123"]`) so that when RAG is used, it can filter by visibility.
- **AI Core** in v1:
  - Does **not** call the backend to resolve objectives or scopes.
  - Stores **context_scopes** on the session and passes them through the graph so they are **ready for future RAG use**.

So the **intended** design is:

- Curriculum structure (which lesson, which objectives) and visibility (which scopes) are decided **outside** AI Core.
- AI Core receives them and, when RAG is wired in, should use **context_scopes** (and possibly lesson/objective) to restrict which content is retrieved.

### 4.2 “Book Contains 5 Topics → Lessons/Objectives Extracted”

The specs (Doc 1–5) do **not** describe:

- Automatic extraction of “lessons” or “objectives” from an ingested book (e.g. “chapter = lesson”, “section = objective”).
- A content model where documents or chunks are tagged with lesson_id/objective_id inside AI Core.

So today:

- **Extraction** of lessons/objectives from a book (e.g. “this book has 5 topics → 5 lessons”) would have to happen **outside** AI Core (e.g. in the backend or an ingestion pipeline that tags documents/chunks and then the backend maps “lesson X” to “documents/chunks with tag X”).
- **Storing** “which knowledge base book to use” could be done by:
  - The caller sending **context_scopes** (and optionally filters) that align with how that book’s content is tagged (e.g. visibility or tags), and
  - AI Core eventually using those scopes (and optionally lesson/objective) when calling search/chat.

That “eventually” part is the missing link in the tutor.

## 5. Summary: Current vs Desired Flow

### 5.1 Current Flow

```
┌─────────────────────────────────────────────────────────────────────────────┐
│ CLIENT (Backend/Nuxt)                                                        │
│ - Decides lesson (e.g. from curriculum).                                     │
│ - Resolves objective_ids (and optionally plain-English objectives).          │
│ - Resolves context_scopes (OU/school/class for visibility).                   │
└───────────────────────────────┬─────────────────────────────────────────────┘
                                │ POST /v1/tutor/start
                                │ { lesson_id, objective_ids or objectives, context_scopes, ... }
                                ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│ AI CORE — Tutor                                                              │
│ - Creates TutorSession (lesson_id, objective_ids, context_scopes stored).    │
│ - Creates one ObjectiveState per objective_id (NOT_STARTED).                  │
│ - Runs teaching loop. Response generation uses LLM + action plan +           │
│   lesson/objective labels only. No search, no RAG. context_scopes unused.     │
└─────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────┐
│ AI CORE — Ingestion & RAG (separate from tutor)                               │
│ - Documents/chunks ingested (e.g. book PDF → chunks with embeddings).        │
│ - /v1/search and /v1/chat use tenant + roles + optional filters.              │
│ - No lesson_id/objective_id in search; no tutor → search/chat call.            │
└─────────────────────────────────────────────────────────────────────────────┘
```

So: **lessons/objectives are caller-defined (or LLM-generated from topic). Ingested documents are stored and used only by /v1/search and /v1/chat. The tutor does not reference any ingested book today.**

### 5.2 Desired Flow (When “Book → Lessons/Objectives” and “Tutor Uses Book” Are Implemented)

- **Extraction (outside or extended ingestion):**  
  Some process (backend or ingestion pipeline) decides that “this book has 5 topics” and either:
  - Tags documents/chunks with topic/lesson/objective identifiers, or
  - Exposes a mapping “lesson_id / objective_id → which scopes or document IDs to use.”

- **Caller:**  
  When starting a tutor session, continues to send **lesson_id**, **objective_ids**, and **context_scopes** that match that mapping (e.g. scopes that restrict to that book’s content).

- **AI Core — Tutor:**  
  When generating explanations (e.g. for EXPLAIN_CONCEPT or GIVE_HINT), the tutor would:
  - Call **search** (or **chat**) with:
    - Same **tenant_id** and **roles** as the session,
    - **context_scopes** from the session (so only that book’s content is visible),
    - Optional **filters** by source_type/tags if the book is tagged that way,
    - **query** derived from the current objective (e.g. objective title or a short prompt).
  - Use the retrieved chunks as context for the LLM so the tutor’s answer is grounded in the ingested book.

That would connect “which knowledge base book” (via context_scopes + optional filters) to “what the tutor says.” Right now only the first part (storing context_scopes and lesson/objective IDs) is in place; the tutor does not yet call RAG.

## 6. Where Things Are in Code

| Concern | Where it is | Notes |
|--------|-------------|--------|
| Lesson/objective **input** | `app/api/v1/tutor.py` → `StartTutorSessionRequest` (lesson_id, objective_ids, objectives) | Caller-provided only. |
| **Resolving** plain-English objectives to IDs | `app/api/v1/tutor.py` → `_resolve_objectives()` | Slugified IDs; no lookup in DB or docs. |
| **Storing** lesson/objectives/scopes on session | `app/tutor/graph_nodes.py` → `load_session_and_profile`; `app/db/models.py` → TutorSession | lesson_id, objective_ids, context_scopes persisted. |
| **Using** lesson/objective in prompts | `app/tutor/thinking_loop_nodes.py` → `node_generate_tutor_response`; `app/tutor/response_generation.py` | lesson_context, objective_context (labels, state) passed to LLM. No RAG. |
| **Using** context_scopes for RAG | — | Not implemented. context_scopes only stored and passed through context. |
| **Ingested documents** | `app/db/models.py` (Document, Chunk); ingestion + search + /v1/chat | No lesson/objective fields. Used only by /v1/search and /v1/chat. |
| **Search filters** | `app/models/schemas.py` (SearchRequest.filters); `app/services/search_service.py` | sourceType, subject, grade, tags. No lesson_id/objective_id. |

## 7. Conclusion

- **How lessons and objectives are created:** They are **not** created or extracted from ingested content inside AI Core. They are **supplied by the client** (or, in the progress-report flow, **generated by the tutor’s LLM** from a topic). Session and objective states are created from these IDs only.
- **How they relate to ingested documents:** Today they **don’t**. Documents/chunks have no lesson/objective fields. Any mapping “book → topics → lessons/objectives” would be in the backend or in tags/source_type; the caller would send the right lesson_id, objective_ids, and context_scopes.
- **How the tutor knows which knowledge base to use:** It **doesn’t** yet. context_scopes are stored for “RAG visibility filtering” but the tutor never calls search or chat, so it never references an ingested book. To implement that, the tutor would need to call the search (or chat) service with the session’s tenant_id, roles, context_scopes, and a query derived from the current objective, and use the returned chunks in response generation.

This matches the client/spec v1: caller supplies objectives and scopes; AI Core does not resolve them from BE; RAG integration for the tutor is left for a later step, with context_scopes and session shape already prepared for it.
